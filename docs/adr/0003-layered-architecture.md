# 0003 - 采用分层架构设计

## 状态

已接受

## 上下文

Kanban Board 项目需要一个清晰的代码组织结构，以支持可维护性和可扩展性。随着功能的增加，原始的单体代码结构变得难以维护。

### 问题分析

- 控制器直接访问数据库，职责不清晰
- 业务逻辑分散在多个文件中
- 缺乏统一的错误处理机制
- 代码重复率较高

## 决策

采用四层分层架构：

```
┌─────────────────────────────────────────────────────────────┐
│  Presentation Layer (表现层)                                │
│  ├── Routes: 路由定义和请求分发                              │
│  └── Controllers: 请求处理和响应                            │
├─────────────────────────────────────────────────────────────┤
│  Service Layer (服务层)                                     │
│  ├── 业务逻辑封装                                           │
│  ├── 验证规则实现                                           │
│  └── 状态转换管理                                           │
├─────────────────────────────────────────────────────────────┤
│  Repository Layer (仓储层)                                  │
│  ├── 数据访问抽象接口                                       │
│  └── 具体数据库实现                                         │
├─────────────────────────────────────────────────────────────┤
│  Data Layer (数据层)                                        │
│  ├── 数据模型定义                                           │
│  └── 数据库配置                                             │
└─────────────────────────────────────────────────────────────┘
```

## 理由

1. **关注点分离**：每层职责明确，便于理解和维护
2. **可测试性**：各层可独立进行单元测试
3. **可扩展性**：便于替换具体实现（如更换数据库）
4. **代码复用**：公共逻辑提取到服务层和工具函数
5. **错误集中处理**：统一的错误处理机制提高一致性

## 影响

### 文件结构变化

| 层级 | 目录 | 新增/修改文件 |
|------|------|-------------|
| 表现层 | `src/controllers/` | `cardController.ts`, `userController.ts` |
| 表现层 | `src/routes/` | `cardRoutes.ts`, `userRoutes.ts` |
| 服务层 | `src/services/` | `CardService.ts`, `UserService.ts` |
| 仓储层 | `src/repositories/` | `CardRepository.ts`, `UserRepository.ts`, `SQLiteCardRepository.ts`, `SQLiteUserRepository.ts` |
| 数据层 | `src/models/` | `Card.ts`, `User.ts` |
| 中间件 | `src/middleware/` | `errorHandler.ts`, `logger.ts` |
| 工具 | `src/utils/` | `request.ts` |
| 错误定义 | `src/errors/` | `index.ts` |

### 依赖关系

```
Routes → Controllers → Services → Repositories → Database
```

## 替代方案

- **单体架构**：简单但难以维护
- **六边形架构**：更复杂但更灵活（对于当前项目规模过于复杂）
- **事件驱动架构**：适合异步场景但增加复杂度
