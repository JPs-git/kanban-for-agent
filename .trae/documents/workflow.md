# 软件开发迭代工作流 v2（Agent 专用）

## 1. 工作流程概述

工作流 v2 专为 Agent 设计，采用规范的迭代模式，包含需求分析、设计、开发、测试和部署等完整环节。通过明确的流程步骤、详细的文档生成和严格的质量保证，确保产品从需求到交付的全过程规范化、透明化和高效执行。

## 2. 工作流程步骤

### 2.1 产品需求文档（PRD）制定

- **责任角色**：产品经理 Agent
- **具体任务**：
  - 根据产品规划和用户反馈分析需求
  - 编写详细的产品需求文档（PRD）
  - 明确功能需求、非功能需求、用户故事和验收标准
  - 组织相关方评审 PRD，确保需求明确且无歧义
  - 最终确定并发布 PRD

### 2.2 UI 设计文档输出

- **责任角色**：UI 设计师 Agent
- **具体任务**：
  - 研读产品需求文档（PRD）
  - 设计产品界面原型和交互流程
  - 制定设计规范和组件库
  - 输出 UI 设计文档，包括视觉设计、交互设计和用户体验说明
  - 组织设计评审会议，收集反馈并迭代设计
  - 最终确定并发布 UI 设计文档

### 2.3 开发计划表制定

#### 2.3.1 前端开发计划表

- **责任角色**：前端架构 Agent(frontend-architect)
- **具体任务**：

  - 分析产品需求文档（PRD）和 UI 设计文档
  - 进行需求拆解和技术方案设计
  - 制定详细的开发计划, 并输出开发计划表

#### 2.3.2 后端开发计划表

- **责任角色**：后端架构 Agent(backend-architect)
- **具体任务**：
  - 分析产品需求文档（PRD）和 UI 设计文档
  - 进行需求拆解和技术方案设计
  - 输出接口文档，包括 API 接口定义、数据模型和错误处理机制

### 2.4 开发实现

#### 2.4.1 前端开发实现

- **责任角色**：前端架构 Agent(frontend-architect)
- **具体任务**：
  - 按照计划表实现功能
  - 编写单元测试, 需要保证不阻塞当前命令行
  - 执行`npm run test:unit` 运行测试并确保通过
  - 计划表上的每一个功能完成后提交一次代码
  - 确认接口文档，确保前端功能与后端接口兼容
  - 确认设计文档, 确保前端界面符合设计规范

#### 2.4.2 后端开发实现

- **责任角色**：后端架构 Agent(backend-architect)
- **具体任务**：
  - 按照计划表实现功能
  - 编写单元测试, 需要保证不阻塞当前命令行
  - 执行`npm run test:unit` 运行测试并确保通过
  - 计划表上的每一个功能完成后提交一次代码
  - 与前端开发人员合作，确保后端功能与前端接口兼容

### 2.5 端到端测试与测试报告生成

- **责任角色**：端到端测试 Agent(e2e-test-expert)
- **具体任务**：
  - 根据 PRD 和设计文档制定测试计划和测试用例
  - 执行端到端测试，验证功能完整性和用户体验
  - 记录测试过程中发现的问题和缺陷
  - 分析测试结果，生成详细的测试报告
  - 提交测试报告给开发团队和产品团队

### 2.6 Bug 修复

- **责任角色**：前端架构 Agent(frontend-architect) 后端架构 Agent(backend-architect)
- **具体任务**：
  - 分析测试报告中的问题和缺陷
  - 按照优先级排序，制定 bug 修复计划
  - 实施 bug 修复，确保问题得到有效解决
  - 进行代码审查，确保修复质量
  - 修复完成后，通知测试人员进行复测

### 2.7 Bug 验证

- **责任角色**：测试团队 Agent
- **具体任务**：
  - 接收开发人员的 bug 修复通知
  - 针对修复的 bug 执行复测
  - 验证 bug 是否完全修复
  - 记录验证结果，形成验证报告
  - 如果 bug 未完全修复，通知开发人员重新修复

### 2.8 Bug 修复循环

- **责任角色**：开发团队和测试团队 Agent
- **具体任务**：
  - 重复执行步骤 2.6 和步骤 2.7
  - 持续修复和验证 bug，直到所有问题得到解决
  - 定期召开 bug 修复进展会议，确保修复工作顺利进行
  - 记录修复过程中的经验教训，为后续迭代提供参考

### 2.9 用户验收

- **责任角色**：产品经理 Agent
- **具体任务**：
  - 确认所有 bug 已修复，功能符合 PRD 要求
  - 组织用户验收测试（UAT）
  - 收集用户反馈并进行必要的调整
  - 最终确认产品符合验收标准
  - 发布验收报告，准备产品上线

### 2.10 部署与发布

- **责任角色**：前端架构 Agent(frontend-architect)
- **具体任务**：
  - 运行 `git push`命令，将本地代码推送到远程仓库, 如果远程未创建分支, 则需要先创建分支, 例如 `git push --set-upstream origin v0.5`
  - 修复所有发现的问题, 并完成推送

### 2.11 迭代回顾

- **责任角色**：全体团队 Agent
- **具体任务**：
  - 回顾迭代过程和结果
  - 分析成功经验和问题
  - 提出改进建议
  - 制定下一个迭代计划
  - 输出迭代回顾文档，包括成功经验和问题、改进建议和下一个迭代计划

## 3. 文档管理

### 3.1 文档规范

1. 所有文档必须符合 Markdown 格式
2. 所有迭代中生成的文档按照迭代版本号存放在 `./.trae/documents/v[版本号]` 目录下
3. 原则上当前版本的迭代文档应该包含上一个版本的迭代文档的所有内容，如果没有变化, 应该添加说明查看上一个版本的迭代文档,并附带链接
4. 每个迭代需要生成完整的项目文档，包括 PRD、设计文档、测试报告等

## 4. 测试环境配置

- **数据库隔离**：测试环境使用单独的数据库（如`kanban-test`），避免影响开发和生产数据
- **环境变量配置**：通过`NODE_ENV`环境变量区分开发、测试和生产环境
- **测试数据管理**：测试前准备测试数据，测试后清理测试数据，确保测试环境的一致性
- **CI/CD 集成**：在 CI/CD 流程中自动设置测试环境，确保测试的可靠性和一致性
